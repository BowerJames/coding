- **Ingest**: Added [Type safety in Python](/python/type-safety.md) (`type: python`) — directive "type safety is critical; preferred type checker is mypy in strict mode." Four-rule policy (strict mode; annotate every function; avoid `Any` by receiving unknown data as `object` and **narrowing** it; `# type: ignore` justified, preferably unneeded). Safety hierarchy `Any` < `cast` < runtime-narrow (`isinstance`/`TypeGuard`/`match`, verified) — `cast` is a justified last resort. Boundary technique: `json.loads()` / `httpx` `.json()` return `Any`; bind with explicit `: object` to kill contagion, then narrow. Minimum `pyproject.toml` setup (`[tool.mypy] strict = true`) flagged as a floor not a ceiling, with recommended extras `disallow_any_explicit` / `warn_redundant_casts`. Preferred data containers: **dataclasses or pydantic** (pydantic narrows at construction; dataclass for already-typed/internal values; `TypedDict` for dict shapes). Noted Python-specific (Rust/TS get type safety from the compiler, so no general `practice` page). Added reciprocal `# See also` link in [Fault tolerance](/error-handling/fault-tolerance.md).
- **Ingest**: Added [Fault tolerance](/error-handling/fault-tolerance.md) (`type: practice`) — the error-handling *policy* layer: applications survive by default and fail at the narrowest scope (fallback → fail the request → crash only as a last resort); every failure is logged at `ERROR` because it means an assumption of the application broke (external-dependency / data-model / internal-state). Includes the REST-`500`-not-crash example, a blast-radius decision table, and the `ERROR`-vs-`WARNING` boundary (a recovered-via-fallback failure is still `ERROR`; `WARNING` is only for expected soft-handling like retries). Directive: "fault-tolerant apps; all failures logged as errors so devs are alerted (wrong data model or failing dependency); strive not to crash, use reasonable fallbacks (REST returns `500`, not a crash)."
